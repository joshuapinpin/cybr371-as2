# OAuth 2.0 / OIDC Security Best Practices

Read this before you start. Each section corresponds to one task.

---

## 1. Algorithm confusion (JWT alg claim)

Never let a token tell you how to verify it. The `alg` header is attacker-controlled. Setting `alg: none` instructs most JWT libraries to skip signature verification entirely — any payload is accepted without a key.

**Rule:** Always hardcode the expected algorithm. Never read `alg` from the token header.

- RFC 8725 §2.1 — "Weak Algorithms"
- OWASP A02:2021 — Cryptographic Failures

---

## 2. Audience validation (aud claim)

The `aud` claim names the intended recipient of a token. Without checking it, a token issued for a different application (e.g. another client sharing the same IdP) can be accepted here. This enables cross-client token confusion attacks.

**Rule:** After decoding an ID token, verify that `aud` exactly matches your own `client_id`. Reject any token where it does not.

- OpenID Connect Core §3.1.3.7
- OWASP A07:2021 — Identification and Authentication Failures

---

## 3. Nonce binding

The nonce is generated by the client at login, stored in the session, sent to the IdP, and embedded in the ID token. Verifying it on receipt proves that this token was issued in direct response to this login request — not replayed from a previous session.

**Rule:** Compare the `nonce` claim in the received ID token to the value stored in the session. Reject the token if they do not match or if `nonce` is absent.

- OpenID Connect Core §3.1.2.1, §3.1.3.7 (nonce)
- OWASP A07:2021 — Identification and Authentication Failures

---

## 4. State parameter (CSRF protection)

The `state` parameter is the CSRF token for the OAuth callback. The client generates a random value, stores it in the session, and includes it in the authorization request. The IdP returns it unchanged in the redirect to the callback. If the returned value does not match the session value, the callback was not initiated by this browser session.

**Rule:** Validate the `state` parameter on every callback. Reject missing or mismatched values. This prevents an attacker from tricking a victim's browser into completing the attacker's authorization flow.

- RFC 6749 §10.12 — Cross-Site Request Forgery
- OWASP A01:2021 — Broken Access Control

---

## 5. Scope enforcement

The `scope` claim in an access token lists the operations the token is permitted to perform. A valid signature and unexpired token is not sufficient — the token must also be authorized for the specific operation being requested.

**Rule:** Check that the required scope is present in the token's `scope` claim before returning protected data. Treat a missing scope claim as insufficient authorization.

- RFC 6749 §3.3 — Access Token Scope
- OWASP A01:2021 — Broken Access Control, API3

---

## 6. PKCE code verifier

PKCE (Proof Key for Code Exchange) binds an authorization code to the specific client session that requested it. The client generates a `code_verifier`, stores it, and sends a derived `code_challenge` in the authorization request. When exchanging the code for tokens, the client must send the original `code_verifier`. Without it, an attacker who intercepts the authorization code can exchange it for tokens themselves.

**Rule:** Retrieve the `code_verifier` from the session and include it in every token exchange request.

- RFC 7636 — PKCE
- OWASP A02:2021 — Cryptographic Failures

---

## 7. PKCE S256 method

With `code_challenge_method=plain`, the challenge equals the verifier — it travels in the front-channel redirect URL and is therefore observable by anyone who can see the URL (browser history, proxy logs, referrer headers). S256 hashes the verifier before sending it, making the challenge safe to transmit publicly.

**Rule:** Always use `code_challenge_method=S256`. Compute `code_challenge = BASE64URL(SHA256(code_verifier))`.

- RFC 7636 §4.2 — Client Creates the Code Challenge
- OWASP A02:2021 — Cryptographic Failures

---

## 8. Open redirect

If your application redirects the user after login to a URL supplied in the request (e.g. `?next=/dashboard`), an attacker can set that URL to an external site (`?next=https://evil.com`). The victim sees the legitimate login page, authenticates successfully, and is then sent to the attacker's site — a phishing setup.

**Rule:** Before redirecting, validate that the target URL is a same-origin relative path. Reject absolute URLs and protocol-relative URLs (those starting with `//`, which browsers resolve as external).

- OWASP A01:2021 — Broken Access Control (Open Redirect)
